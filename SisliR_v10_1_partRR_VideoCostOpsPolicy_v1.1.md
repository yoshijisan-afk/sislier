# Part RR — 動画生成コスト運用ポリシー（v10.1追加）
## Higgsfield サブスクリプション移行 + 並行ノウハウ蓄積戦略

> **位置づけ**: 既存設計（Part G: VideoGeneratorRouter、Part QQ: ABテストエンジン、
> Part DD: OrchestrationLoop）に対する**運用ポリシーの追加のみ**。
> コード・スキーマ・アーキテクチャの変更は発生しない。
> 変更対象は「Higgsfieldの課金方式」と「クレジット管理・モニタリング運用」のみ。

---

## RR.1 背景

Part P.3のコスト試算では、Higgsfield動画生成を**従量課金（¥1,500/本）**前提としていた。
Higgsfieldには月額サブスクリプションプランが存在し、クレジット制で運用すれば
**原価を固定費化し、件数増加時のスケールメリットを活かせる**。

国内向けサービスとして、初期はHiggsfield（高品質・実績あるシネマティック生成）を
メインエンジンとしつつ、生成結果と効果データ（視聴完了率・CVR）を
WinnerPatternService（Part QQ.9.1）に蓄積し、将来的にOpenCut MCP + GSAP4ベースの
自社カメラワーク生成へ段階的に移行する。

## RR.2 Higgsfieldプラン選定方針

| Phase | 対象プラン | 月額目安 | クレジット/月 | 想定生成本数 | 対応テナント規模 |
|-------|-----------|---------|--------------|-------------|-----------------|
| Phase 0-1 | Basic または Pro | $9〜$29 | 150〜600 | 約3〜20本 | Starter〜Growth |
| Phase 2 | Ultimate | $49 | 1,200〜1,500 | 約20〜40本 | Growth〜Premium |
| Phase 3 | Pro/Basicへダウングレード | $9〜$29 | — | 補助用途のみ | 自社エンジンが主力化後 |

**選定基準（毎月見直し）**:
- 当月のActive Growth/Premiumテナント数 × 想定1〜2本/月 で必要クレジットを算出
- 1本あたり消費クレジットはモデル・解像度・尺により30〜60クレジットの幅があるため、
  保守的に60クレジット/本で見積もる
- 算出値が現プランのクレジット上限の70%を超えたら、次プランへのアップグレードを検討

## RR.3 クレジット監視運用（CostMonitor拡張・運用ルールのみ）

既存の `lib/cost/CostMonitor` に、Higgsfieldクレジット残量チェックを
**定期実行タスクとして追加**する（新規モジュール開発は不要、cron設定の追加のみ）。

```
監視項目: Higgsfieldダッシュボードのクレジット残量（手動 or API取得）
記録先:   api_cost_logs（既存テーブル。provider='higgsfield', task='credit_balance'）
頻度:     日次（DD.9のcron設定に追加）

アラート閾値:
  残量 < プラン上限の30% → Slack/メール通知（BB.6既存アラート経路を流用）
  残量 < プラン上限の10% → VIDEO_PROVIDER_PRIORITY を一時的に
                          'opencut,ffmpeg' 優先に切替（環境変数のみ変更）
  残量 = 0               → 自動フォールバック（Part G既存ロジックがそのまま機能）
```

**重要**: VideoGeneratorRouterのフォールバック実装は変更不要。
`VIDEO_PROVIDER_PRIORITY` 環境変数の並び替えのみで挙動が切り替わる
（既存設計どおり）。

## RR.4 並行学習（ノウハウ蓄積）方針

Higgsfieldで生成した動画は、既存のQQ.7（VideoAbRouter / 視聴完了率計測）と
QQ.9.1（WinnerPatternService）にそのまま流す。**新規スキーマ・新規ロジックは不要。**

```
短期（Phase 0-1）:
  Higgsfieldをメインエンジンとして運用
  → 全生成動画をab_video_styles等の既存テーブルに記録（QQ.2.3）

中期（Phase 2）:
  WinnerPatternServiceに蓄積された「物件種別×演出パターン×成果」データを
  四半期ごとにレビュー
  → 「どの物件種別でどんなカメラワーク・トランジション・尺が成果につながるか」
    のパターンを人間がドキュメント化（docs/ops/video-patterns.md 等）

長期（Phase 3）:
  蓄積パターンを元に、OpenCut MCP + GSAP4 + CameraSystem（既存）で
  再現可能な演出テンプレートを拡充
  → Higgsfieldは「テンプレートでは対応しきれない高単価受注案件（トラックB）」
    向けの補助エンジンとして位置づけを縮小
```

## RR.6 ADRログ

| ADR# | 決定内容 | 採択理由 | 代替案（不採用） |
|------|---------|---------|-----------------|
| ADR-168 | Higgsfieldを従量課金から月額サブスク（Pro/Ultimate）へ移行し、クレジット残量を日次監視。残量30%未満でOpenCutへのフォールバック優先度を上げる | 件数増加時のスケールメリット獲得とコスト固定化。既存のVideoGeneratorRouterフォールバック機構を活用するだけで実現可能、追加実装コストゼロ | 従量課金継続（件数増加で原価が線形増加し粗利を圧迫） |
| ADR-169 | Higgsfield生成動画の効果データを既存のWinnerPatternService（QQ.9.1）に蓄積し、四半期ごとに人間がレビューして演出パターンをドキュメント化する | 「自社AI学習」を新規MLパイプラインとして構築するのは過剰投資。既存A/Bテスト基盤の延長で十分にノウハウ資産化できる | 専用の動画生成モデルをファインチューニング（開発コスト・データ量ともに時期尚早） |

