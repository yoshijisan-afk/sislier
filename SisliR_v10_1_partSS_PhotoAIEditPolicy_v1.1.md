# Part SS — 生成AI画像編集オプション機能 運用ポリシー（v10.1追加）
## ChatGPT 4o / Gemini API連携 + プロンプトエンジン活用

> **位置づけ**: 既存設計（Part F: PropertyIntakeAgent Step2写真処理、
> ADR-047: 写真補正=sharp）に対する**追加オプション機能の運用ポリシー**。
> sharpによる基本補正（Step2・全件自動・コスト¥0）は変更しない。
> 生成AI画像編集は**別レイヤーの新規オプション機能**として、
> 物件詳細の写真一覧から個別選択・オプトインで実行する。

---

## SS.1 背景・処理レイヤーの整理

「写真のレタッチ」には2つの異なるレイヤーが存在し、混同しないこと。

| レイヤー | 内容 | エンジン | コスト | 既存設計 |
|---------|------|---------|--------|---------|
| ① 基本補正 | 明るさ・コントラスト・ホワイトバランス・WebP変換 | sharp | ¥0 | ADR-047確定済み・変更なし |
| ② 生成AI編集 | 背景差し替え・空の演出・3Dアイソメ化・イラスト化等 | ChatGPT 4o / Gemini（外部API） | 従量課金（新規） | **本Partで追加** |

Claude（claude-sonnet-4 / haiku）は画像を**生成しない**（理解・分析・判定のみ）ため、
②のレイヤーをClaude APIで代替することはできない。①は引き続きsharpが最適。

---

## SS.2 機能概要

物件詳細ページの写真一覧から、写真ごとに「生成AI編集」をオプトインで実行できる
機能を追加する。

```
物件詳細 > 写真一覧
  各写真サムネイルに「✨ AI編集」ボタンを表示
       ↓ クリック
  ARCH PROMPT エンジン（既存アップロード資産）が
  写真の特徴・スタイル選択（カラー/白黒/3Dアイソメ/イラスト風等）に基づき
  プロンプトを自動生成
       ↓
  プロンプトを画像と共に ChatGPT 4o (gpt-image-1) API へ送信
       ↓
  編集結果を確認 → 採用 / 再生成 / 破棄
       ↓ 採用時のみ
  R2に保存（差し替えではなく追加バリアント: 06_AiEdited/）
```

## SS.3 コスト構造とプラン制限

| プラン | 生成AI編集利用可否 | 月間上限 |
|--------|-------------------|---------|
| Starter | ✕ 不可 | - |
| Growth | ◯ 利用可 | 月10枚まで |
| Premium | ◯ 利用可 | 月50枚まで |
| Enterprise | ◯ 利用可 | 要相談（従量課金オプション） |

- gpt-image-1の1枚あたりコストは解像度・編集範囲により変動するため、
  保守的に1枚あたり¥30〜100の幅で見積もり、`api_cost_logs`に
  `provider='openai', task='photo_ai_edit'`として記録する。
- 月間上限到達時はUI上で「✨ AI編集」ボタンを非アクティブ化し、
  「今月の上限に達しました」を表示（Higgsfieldクレジット枯渇時の
  UI挙動・Part RR RR.3と同様のパターンを流用）。

## SS.4 実行ルール（コスト最適化）

1. **デフォルトOFF・物件登録時の自動実行には絶対に含めない**
   PropertyIntakeAgent Step2（process_photos / generate_thumbnail）の
   自動フローには本機能を追加しない。ユーザーが写真一覧から
   個別に選択した場合のみ実行する。

2. **べき等性: 生成結果はキャッシュし、同一写真+同一スタイル選択の
   再リクエストは確認ダイアログを出す**（「既に生成済みです。
   再生成すると追加でクレジットを消費します」）。

3. **プロンプト生成自体はAPI課金なし**
   ARCH PROMPT エンジンはローカル（クライアントサイドJS）で
   プロンプト文字列を構築するため、この段階ではコストが発生しない。
   課金が発生するのは画像編集APIへの送信時のみ。

4. **採用前のプレビューは縮小解像度で生成**し、ユーザーが
   「採用」を押した場合のみ本解像度で再生成する（2段階生成により、
   不採用案へのフル解像度コストを回避）。

## SS.5 ADRログ

| ADR# | 決定内容 | 採択理由 | 代替案（不採用） |
|------|---------|---------|-----------------|
| ADR-170 | 生成AI画像編集（背景差し替え・3D化等）はChatGPT 4o (gpt-image-1) / Gemini APIを使用し、sharpによる基本補正（ADR-047）とは別レイヤーの機能として実装する | Claude APIは画像を生成できないため、この用途には不適。sharpは数値補正専用で生成編集は不可。両者を置き換えるのではなく、レイヤーを分離して並存させるのが最も整合的 | Claude APIで画像編集を代替（技術的に不可能） / sharpを生成AI処理に置き換え（基本補正の精度・速度・コストが悪化） |
| ADR-171 | 生成AI画像編集はデフォルトOFFのオプトイン機能とし、物件詳細の写真一覧から写真ごとに個別選択・実行する。PropertyIntakeAgentの自動フロー（Step2）には含めない | 自動一括実行はHiggsfield動画と同様にコスト爆発リスクが高い。実際に使われる写真のみ課金対象にすることで原価を最小化できる | 全写真への自動適用（コスト管理不能） |
| ADR-172 | 生成AI画像編集はプラン別の月間枚数上限（Growth:10枚, Premium:50枚）で制御し、`api_cost_logs`に`provider='openai'`として記録する | Higgsfieldクレジット監視（Part RR）と同じ枠組みでコスト管理を一元化できる | 完全従量課金をデフォルトに（予期せぬ高額請求リスク） |

## SS.6 実装作業（該当する場合のみ）

```
1. 新規APIエンドポイント: POST /api/photos/[id]/ai-edit
   - 入力: 写真URL、スタイル選択（ARCH PROMPTエンジンのオプション）
   - 内部処理: ARCH PROMPTロジックでプロンプト生成 → OpenAI API呼び出し
   - 既存の OpenTelemetry スパン規約に従う

2. 環境変数追加:
   OPENAI_API_KEY=
   PHOTO_AI_EDIT_MONTHLY_LIMIT_GROWTH=10
   PHOTO_AI_EDIT_MONTHLY_LIMIT_PREMIUM=50

3. DBスキーマ追加: photo_ai_edit_logs テーブル
   (property_id, photo_id, style, status, cost_jpy, created_at)
   ※既存 api_cost_logs と合わせて使用

4. UIコンポーネント追加: 物件詳細 > 写真一覧 > 「✨ AI編集」ボタン
   （ARCH PROMPTエンジンのスタイル選択UIを流用）

5. 既存のPart F（PropertyIntakeAgent）コード変更: なし
```
